# Dual-Backend Configuration Guide

> How to select and configure the persistence backend for `mcp_daemon_engine`.

## Overview

`mcp_daemon_engine` supports two persistence backends for its 4 metadata entities
(`MCPFunction`, `MCPModule`, `MCPSetting`, `MCPFunctionCall`):

| Backend | `DB_BACKEND` | When to use |
| --- | --- | --- |
| DynamoDB | `dynamodb` (default) | AWS-native deployments, serverless, existing DynamoDB infrastructure |
| PostgreSQL | `postgresql` | Self-hosted, RDS, multi-tenant shared DB, complex query patterns |

**S3, the MCP runtime (SSE/stdio/JSON-RPC), the external-MCP proxy, and dynamic
tool-module loading are NOT backend-selectable.** They work identically under both
`DB_BACKEND` values.

## Selection

Set `db_backend` in the configuration setting dict passed to `Config.initialize()`:

```python
setting = {
    "db_backend": "postgresql",  # or "dynamodb" (default)
    # ... other settings
}
Config.initialize(logger, setting)
```

Or via environment variable in the gateway `routes.yaml` setting block.

## DynamoDB Mode

Required settings:
- `region_name`
- `aws_access_key_id`
- `aws_secret_access_key`

Optional:
- `initialize_tables: true` — creates the 4 DynamoDB tables on startup if they don't exist.

Tables (PAY_PER_REQUEST billing):
- `mcp-functions` — hash: `partition_key`, range: `name`, LSI: `mcp_type`
- `mcp-modules` — hash: `partition_key`, range: `module_name`, LSI: `package_name`
- `mcp-settings` — hash: `partition_key`, range: `setting_id`
- `mcp-function_calls` — hash: `partition_key`, range: `mcp_function_call_uuid`, LSI: `mcp_type`, `name`, `updated_at`

Cache: `@method_cache` is active on model getters. Cascading cache purge is active.

## PostgreSQL Mode

Required settings:
- `db_host`, `db_port`, `db_user`, `db_password`, `db_schema`

Optional:
- `initialize_tables: true` — runs `Base.metadata.create_all(checkfirst=True)` on startup.

Install PG dependencies:
```bash
pip install mcp-daemon-engine[postgresql]
```

Tables (created by Alembic migrations or `create_all`):
- `mcp_functions` — PK: `(partition_key, name)`, index: `(partition_key, mcp_type)`
- `mcp_modules` — PK: `(partition_key, module_name)`, index: `(partition_key, package_name)`
- `mcp_settings` — PK: `(partition_key, setting_id)`
- `mcp_function_calls` — PK: `(partition_key, mcp_function_call_uuid)`, indexes: `(partition_key, mcp_type)`, `(partition_key, name)`, `(partition_key, updated_at)`

Cache: PG repositories do not use `@method_cache` today. `CACHE_ENTITY_CONFIG_POSTGRESQL`
is empty. The `purge_entity_cascading_cache` side effect still fires after writes
(it no-ops on PG since the cache config is empty).

## S3 in PostgreSQL Mode

**S3 stays initialized unconditionally** when `funct_bucket_name` is set, even in PG
mode. This diverges from `rfq_engine`'s "AWS fully optional in PG mode" rule because
`mcp_daemon_engine` uses S3 for:
1. MCP package upload/download (mandatory runtime feature).
2. Large-content offload for `mcp_function_call`.

When AWS credentials are absent, S3 falls back to the default credential chain.

Cognito and Lambda clients remain conditional on credentials + auth provider in both modes.

## S3 Content Offload Divergence

| Behavior | DynamoDB | PostgreSQL |
| --- | --- | --- |
| Auto-offload on size limit | Yes (400KB DynamoDB item limit) | No (no row size limit) |
| Offload when `content_in_s3=True` | Yes | Yes (explicit caller request only) |
| Hydrate content from S3 on read | Yes | Yes |

Both backends support reading `content_in_s3=True` rows — the PG repository fetches
content from `s3://<bucket>/mcp_content/<uuid>.json` just like DynamoDB does.

## No `uuid-ossp` Extension Required

All primary keys are `String` (not UUID-typed). The `mcp_function_call_uuid` is
generated by `str(uuid.uuid4())` in Python and stored as a string.
